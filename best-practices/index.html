Email best practices:

Blocked images, mobile browsers, and varying user settings are a few of the things to consider when designing or coding an email.

Basic Guidelines

Basic guidelines: Keep the design clean and simple. Use a max-width of 600px, in-line css, ditch the images and all text should be live.

All text should be live, not dead.

Live text is the key to a good HTML email. A high text-to-image ratio will keep your email out of spam folders, and improve readability when images are blocked (only a handful of email clients enable images by default). When text is in an image, consider it dead on arrival. The headlines and calls-to-action especially must be live to maximize open and click rates.

I'm alive! I'm dead.

Full disclosure: it is possible to create live buttons with gradients and rounded corners. BUT—it must be able to work with a plain background and square corners so that it can degrade gracefully in even the worst email clients (talking to you, every version of Outlook since 2007).

Keep it simple.

White space and simplicity in emails have been proven to increase click rates. Emails often have multiple calls-to-action or multiple modules. These should be constructed with a clear hierarchy and room to breathe. There is no limit to available vertical space, and separating elements improves readability. Don't feel pressured to keep every single element "above the fold".

Mobile is a must.

Making emails work on mobile is a requirement, not a goal. Keep your calls-to-action large and fingertip-friendly. Stay within a maximum width of 600 pixels, and use a hierarchy of text sizes and weights for instant readability on a smaller screen.

The more readable an email is, the more successful it will be.

It's tempting to want to try using clever twists on HTML defaults, but be sure to back up any grid breaks, light-on-dark designs, or link color changes with sound logic. Emails that follow visual conventions with simple layouts, dark-on-light designs, and blue links with underlines will be more readable, and consequently more effective.

Design for a situation with blocked images.

The majority of email clients will automatically block images when an email is opened. An email design should be able to make perfect sense even when all the images are replaced with plain boxes and alt-text.

HTML(5) Banners and in-app Ads

Why HTML5?

The demand for HTML5-formatted ads increases as the promise for seamless support across desktop computers and mobile devices is propagated throughout the digital advertising marketplace. While HTML5 offers rich design features, it comes with the need for a different skill set outside traditional email development.

HTML5 is the latest update to the Hypertext Markup Language, that includes new semantic tags for features such as video, audio, canvas and other design features.

Original Standard HTML5:

The STANDARD HTML5 bundle is a baseline unit for simple, quick turnaround creative executions. The level of HTML, CSS and linked scripts require a minimal amount of coding, basic animation and images, typically constrained to a 40k file size or less. The STANDARD HTML5 bundle is optimized for modern cross-browser compatibility and run on mobile devices.

Complex HTML5 for Browser or Mobile devices.

The COMPLEX HTML5 bundle is a baseline unit for more complex creative executions. There is a higher level of HTML, CSS, and Scripting. The COMPLEX HTML5 Banner may require the use of modern HTML5 capabilities, integrated video, canvas elements, JSON, XML, Third-party API integration and is optimized for modern cross-browser compatibility and targeted mobile platforms or devices.

HTML5 provides some of the following benefits:

  • It's based on a well-established format that many authoring tools and developers can produce.
  • It provides a rapid testing infrastructure that all browsers should be able to process without using rendering libraries or wrappers.
  • It preserves HTML standard methods of laying out content such as positioning, layering and opacity.
  • Ads created in HTML5 offer benefits for flexibility and freedom to delvier ads across platforms and mobile devices. In addition, the demand for rich media is driving adoption of HTML5.

HTML5 Ad Distinction

An HTML5 ad is effectively an HTML document or a web page that follows the W3C/WHATWG standards, meaning that web browsers must be able to render the ad like a web page. Preceding all content should be the HTML tag <!DOCTYPE html>. The document should also contain at least <html> and <body> tags.

General HTML5 Display Ads are limited to the following traits:

  • Creative that does not expand out of its initial boundaries
  • All assets should be packages and accounted for the file size.
  • Desktop ads will range around ~40K while in-app ads range from ~2MB.
  • Before developing HTML5 ad creative, be sure to consider the technical requirements of the pertinent vendors and partners for ad delivery (ad server and publishers) — File size, packages, external libraries, coding ClickTags & tracking, backup image.

    HTML5 doesn't have the compression and packaging capabilities as that of Flash, and with high-density displays permeating the market, larger creative assets are necessary in HTML5 to produce crisp visuals.

Code and assets.

HTML, CSS and JavaScript code are typically written in legible text that is highly compressible and therefore should be minified.

Asset compression is increasingly important with high-density displays. To compress assets, use PNG "crushers" and other compressors, leverage formats such as scalable vector graphics (SVG) whenever applicable.

HTML text should be used whenever possible while incorporating fonts that are either stored as part of the ad assets, or delivered dynamically as web fonts. Using text and scalable fonts rather than attempting to bake the text into an image offers significant advantages that improve readability and legibility, reduce image use and improve the accessibility of the HTML5 ad content. Use web-safe fonts (system fonts) whenever possible to reduce file size.

HTML5 Animation:

HTML5 animation can be incorporated into display ads in a number of ways and may not be processed the same way as traditional methods. Certain considerations for performance, execution, and publisher/ad server requirements must be taken into account. The following sections offer recommendations for navigating HTML5 animation options.

Animation Peformance

Creative designers have various animation methods to choose from, but the choice of certain animation elements can overwhelm the CPU and affect ad laod performance. CPU usage can be affected by various animation elements: number of objects animating simultaneously, the efficiency of the code, excessive animated images with transparency, multiple CSS3 effects and complexity of the document's layout.

Consider proper use of JavaScript time controls like, setInterval, setTimeout, requestAnimationFrame; animation loops or DOM element manipulation.

CSS Animations VS. jQuery/JavaScript Animations

jQuery/JS animation offers more control over CSS3 methods. jQuery may be necessary when key animation must be seen on desktop browsers although using jQuery creates extra overhead upon loading JavaScript in conjunction with the ad's code, which results in a delay in the animation play.

CSS(3) animation is encouraged for improved performance because it is native code that can execute quickly and involves less coding, which eases the use of the processor. CSS is your best option over jQuery. Support for CSS3 methods should be verified through feature detection prior to serving.

Unlike Flash, there is no true FPS setting within an HTML5 document, however, there are ways to define Frame per second using a JavaScript variable, but it is not required.

HTML5 Canvas Feature Support Check:

In order to safely use HTML5 Canvas the creative should contain code that checks the browser for support of this feature. The following code sample is a way to check for Canvas support, and if the feature is not supported, then it will fall back to whatever content is contained within the Canvas tag, typically a static JPEG version of the ad:

function canvasSupport(){
return !!document.createElement('canvas').getcontext;
};

STANDARD FLASH

The Flash file size problem:

One of the biggest road bumps in Flash development is what happens when a lovely uncompressed PSD needs to be squeezed into a tiny little 40K SWF. By knowing how much room you have to work with, it becomes easier to design creative to maximize available space and avoid crunchy image compression.

What a standard banner looks like?

Complexity: Standard | Standard Flash Banners are SWF files that are used for displaying images, graphics and type. These ads can contain animations with varying degrees of interactivity and function.

Work Type: New Master assumes the technical production and testing of a newly coded Flash banner template with a solid code foundation for creative refreshes, new versions and modifications.

What fits in 40K?

Image file weight mostly depends on number of colors and amount of detail. The more complex the image is, the heavier it will be. A good rule of thumb is that one banner can accommodate its own pixel dimensions fairly well with little compression. So, a 300x250 banner can take one 300x250 jpeg or equivalent.

If images have fewer colors or are less detailed, more image surface area can be squeezed in. Keep in mind that of the standard banner sizes, it's the 160x600 that has the most surface area to cover, and tends to have the most visual compression problems.

Below are some actual screen sizes of standard banners. Get a feel for the dimensions, so you can more easily determine what can and can't fit inside them.

728x90 Leaderboard

160x600 Skyscraper

300x250 Mid-Page Unit (MPU)

320x50 Mobile (static or animated GIF)

Solving the file size problem (in creative).

Use repeating elements. Flash doesn't care if you use the same image a hundred times; it will only contribute to the file size once. When possible, try to use vector graphics instead of raster, as vectors will be far more lightweight and don't need any compression.

Solving the file size problem (in development).

Many image effects can be recreated in Flash with vector masks, blend modes, and simple filters. Often, reflections can be created manually in Flash as well. The image is brought in as a cropped and optimized jpeg, a vector mask is applied to remove the background, and it's then flipped and masked with a gradient. In desperate situations, symmetrical images can be cropped in half and mirrored in Flash to create a whole.

The details of animating:

Animation notes from creatives are necessary for a smooth development process. It doesn't matter if you say "make this flop around like a fish, sit for a few seconds, and then fly out to the top right like it's on fire", the more descriptive, the better.

The word "animates" is banned.

"The text animates onto the stage and then the image animates in above it." No. Be more specific. "Fades", "slides", "drops", "reveals", etc. are much more helpful verbs.

Direct the style of the animation.

When a banner has a call-to-action on the final frame, should there be a rollover effect? Do you want a replay button to appear? What kind of look and feel are you going for, sharp and zippy? Soft and dreamy? Fun and bouncy? Start thinking about things like interactions and motion style.

Remember the medium.

Flash is not After Effects. Complex morphs, 3D effects, piles of independent moving parts, or panning across giant images are at best, difficult to do well, and at worst, nearly impossible. These sorts of animations just aren't appropriate for Flash, and you could end up disappointed by the results.

Add a 1-pixel border.

This applies to all Flash (standard and rich) and static banners. It’s a best practice, and often a vendor requirement, to add a 1-pixel border to an ad. Usually it ends up being a medium gray, but the exact shade can be chosen to complement the creative. Just make sure the color is consistent within each concept.

The border helps to anchor an ad on the page, especially in cases where the colors at the edge of the ad may match the page's background.

RICH MEDIA

The rules of rich media:

With rich media campaigns, there are a lot of new opportunities. IMPORTANT: Ask for specs before development. Know the Ad Serving Platform, requirements, k-weight, dimensions, FPS, dynamic elements and analytics. The campaign’s limiting factor is often going to be the details of the media buy. For efficiency, it's normal to build files to adhere to the strictest specifications of a campaign (a "lowest common denominator" strategy). So, a lot of the projects end up following a set of specific types.

What a Rich Media banner looks like.

Complexity: Standard | Standard Rich Media Assumes advanced technology integration requiring and/or a database or server side code, Flash Media Server, consumption of data from 3rd party sources (API's, RSS) or data capture. Includes in-banner videos, audio, standard expanding panels, video player with reskin.

Work Type: New Master assumes the technical production and testing of a newly coded RichMedia banner template with a solid code foundation for creative refreshes, new versions and modifications.

Secure Ad Serving

DFA Example: Secure pages require secure tags. If the URL of a page starts with https, make sure the URLs in your tags start with https as well. That includes DFA tags and third-party URLs, and any other tags you place on a secure page. Otherwise, your ads may not serve correctly.
Image Example: < img src="https://my.cdn/RichMedia/my_header_image.jpg" / >

What is a secure page? A secure page uses encrypted data to keep communications over the internet secure. Any URL that starts with https uses a secure protocol to encrypt data. Usually it's either an SSL or TLS protocol (a Secure Socket Layer protocol or a Transport Layer Security protocol). As you might guess, the s at the end of https stands for "secure": Hypertext Transfer Protocol Secure.

Survey URLs, third-party impression trackers, and other third-party URLs in a creative must use https when placed on secure sites. This includes, for example, URLs from Dynamic Logic, Insight Express, Evidon, and DoubleVerify. These tags and URLs must be manually modified. Please contact any third-party companies you work with to request secure versions of their tags or URLs.

Rich media introduces new opportunities for engagement.

The most common elements seen in rich media campaigns are polite loads, expanding units, and embedded video. “Polite load” is a method where a lightweight flash file is used to bring in a heavier file once the entire page is loaded. Most rich media formats are based on the concept of polite loading.

It's always good practice to lock down the specs for the inital polite load and secondary loads for the development team.

Rich media UX Considerations:

With every opportunity for user interaction, there is an opportunity to influence user behavior through good design.

Consider "expand" calls-to-action and "close" buttons.

Make it clear to the user what's going to happen when they click your ad. A vague "click to see more" may make it seem like clicking will take them away from the page they're on. "Click to expand" or even "expand to see more" are more specific, and can increase interaction rate. Close buttons must be easily visible and intuitive.

Video players must have basic controls (at least).

Most vendor specs will require play/pause and mute/unmute as the bare minimum for a video player. It’s better to design these controls intentionally than rely on ugly defaults (see a few below—consider yourself warned).

Interactions need to be fully fleshed out.

If you’re asking the user to do something, it has to be dumbed down and extremely obvious. Don’t leave any ambiguity in your instructions. Use specific diagrams, or even little moving cursors and arrows if it’s more complicated. Do the instructions fade out once the user has begun the interaction? What does it look like once the user has succeeded? What if they stop halfway through? Provide visual feedback for every step of the process.

GRAPHICS

Digital file formats:

For people more experienced with print files, the world of web images can get pretty confusing. Clearing up the types, limitations, sizes, and capabilities can help everyone get on the same page. These rules apply not only to static banners, but also to emails and Flash banners, which use these formats to control file size and optimize for web.

JPEG

JPEG is a lossy format that easily compresses to small sizes. If an image is compressed too much, things may start to look “crunchy”.

At decent quality settings, this format is most useful for images with many colors or a lot of detail, especially photographs. Any solid color blocks or vector graphics will probably look like garbage though, even at a higher quality setting.


Hover

PNG-24

PNG (24-bit) is a lossless format with all the capabilities as a JPEG, plus the ability to display large blocks of solid colors very well. This format also supports transparency extremely well, including partial opacity. The only problem is that the file size is also much larger than most other formats.


Hover

GIF/PNG-8

GIF (and the similar PNG 8-bit) images are processed with an interesting color model. The number of unique colors in an image determines the file size, and those colors are indexed in a list. The maximum number of colors is 255 + alpha, so ideally, this file format is used for images that have few colors and not much detail. It also supports transparency, but only if the pixels are fully transparent or fully opaque.

Any areas with visual gradients are usually compressed into very few colors, and can be tweaked through various dithering settings. Different dithering algorithms will scatter the pixels in different patterns, and the ideal setting depends on the image. GIFs are often useful for simple graphics with few colors, like solid-color logos.


Hover

Animated GIF

Animated GIFs are the same format as a regular GIF, so they follow all the same rules, but they have some added considerations. An animated GIF’s file size depends largely on the amount of pixels that change in between frames. If the animation is subtle, it can be optimized much more successfully.

Animated GIFs are at their best when displaying a mostly static image with small moving parts, rather than a slideshow.

Color spaces:

For images in print, color information is usually saved as a combination of cyan, magenta, yellow, and black ink. Whenever images are viewed on a screen however, these colors are being represented as a combination of red, green, and blue light. It’s important to realize the differences in order to prevent conversion issues.

Yes, all digital creative needs to be RGB.

If the colors in a design are delivered in CMYK, there may be major visual issues with inconsistent hues. Many brand-specific style guides give specific RGB hex value equivalents of certain spot colors. Use these colors for any digital projects. Any brand logos or graphics also may need to be color-corrected when converted to RGB.

Look out for common problems.

It might seem obvious, but the most common conversion problems tend to occur in the brightest reds, greens, and blues. RGB has a wider spectrum of colors available than CMYK. When colors go from RGB to CMYK and back to RGB, reds can turn slightly pinkish, and blues and greens get much more dull. Make sure your brightest hues are being represented correctly.

See the rainbow spectrum below, in RGB, and converted to CMYK and back to RGB, respectively.

Below is the same thing, but isolated to red, green, and blue. Keep this in mind especially for logos, which often make use of saturated reds and blues.

GIF and PNG-8 files have limited palettes.

Both GIF files and PNG-8 have 8 bits of storage for red, green, and blue each. This means the total amount of colors available in one of these files is 256 (83). Because of this, an image with more colors must be saved in a different format with (usually) a larger file size. For animated images, which must be GIFs, the color palette will always be forced down to 256 colors.

Image resolution:

There seems to be a lot of confusion on the subject of image resolution in digital applications. Long story short: it’s meaningless.

PPI or DPI?

PPI (pixels per inch) is the usual metric used for image resolution in print. DPI (dots per inch) refers solely to a printer’s ink density settings. Neither matters for digital graphics. While it’s commonly thought that images need to be saved at 72 ppi for digital applications, this is misleading. The 72 ppi setting is simply the default when an image with no resolution settings is opened in Photoshop.

The internet is in pixels, not inches.

Some image formats save resolution and dimension information in special file attributes called metadata. When a JPEG is saved “for Web & Devices” out of Photoshop, this metadata is discarded to save file size. For PNG and GIF files, this metadata doesn’t get saved anyway. Pixel size is the only measurement that gets saved.

One of these kittens was saved at 72 ppi, and the other was saved at 1000 ppi (note: they were saved with a regular "Save As" instead of "Save for Web & Devices", which discards resolution). Both have the same pixel dimensions. Can you tell which is which without downloading and opening them in Photoshop?

“Hi-res” doesn’t apply.

Of course, an asset’s Photoshop documents should be created non-destructively, so that larger elements are embedded as smart objects. The complete asset though, whether it’s an email, Flash storyboard frame, or static banner, will only exist at the correct pixel dimensions with an irrelevant resolution setting.
